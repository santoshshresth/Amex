Overview

Full project framing

This research project is designed as a live fintech intelligence build focused on explainable credit default prediction for regulatory-compliant lending decisions.

Background

Why this project matters

Credit default prediction is one of the most important machine learning applications in financial services because every lending decision depends on estimating whether a borrower is likely to repay. Traditional score-driven approaches are easier to explain, but real borrower behavior is rarely linear, and important signals often emerge through complex interactions across repayment history, utilization patterns, balances, and delinquency behavior.

As credit systems become more data-driven, financial institutions need models that can improve portfolio quality without creating opaque decision pathways. The key delivery challenge is not only stronger prediction, but prediction that remains understandable, auditable, fair-aware, and operationally defensible.

Project Intent

What this initiative is built to achieve

The project uses XGBoost as the primary predictive engine and SHAP as the main explainability layer. The objective is to create a framework that can accurately identify likely defaults while also showing why the model behaves the way it does at both portfolio level and individual case level.

The final delivery package is intended to balance predictive performance, explanation quality, fairness awareness, regulatory alignment, and reproducible engineering practices in one integrated execution stream.

Research Questions

Core decision questions

  • How will tuned XGBoost perform versus Logistic Regression and Decision Tree across the selected credit datasets?
  • How effectively can SHAP make the XGBoost decision process understandable to different stakeholders?
  • What computational overhead is introduced when SHAP is integrated into the pipeline?
  • Can SHAP-based counterfactual guidance provide realistic and actionable improvement directions?
  • Will the explanation layer align with transparency and fairness expectations under GDPR, ECOA, and FHA?
  • Which class imbalance strategy gives the best balance between predictive quality and explanation reliability?
Objectives

Execution goals

  • Design and develop an XGBoost-based machine learning framework for credit default prediction.
  • Evaluate predictive performance through AUC ROC, Precision, Recall, F1 Score, MCC, and Brier Score.
  • Integrate SHAP to interpret model behavior and strengthen transparency for lending decisions.
  • Retain Logistic Regression and Decision Tree as transparent reference models for comparison.
  • Measure explanation stability, fidelity, communicability, and counterfactual usefulness.
  • Embed reproducibility through fixed seeds, version control, modular code structure, and environment consistency.
Project Significance

Business and operational value

Financial institutions often face a tension between strong predictive power and explainable decisioning. This project demonstrates a delivery pathway where portfolio optimization and transparent decision support can operate together rather than as competing goals.

The framework also creates reusable guidance for future AI-led credit systems where performance, fairness checks, and structured explanation must coexist in one operating model.

Scope

Included and excluded boundaries

  • Included: Two public credit datasets, XGBoost, Logistic Regression, Decision Tree, SHAP, optional LIME, class imbalance strategies, fairness review, and compliance mapping.
  • Included: Missing value treatment, feature engineering, outlier control, Bayesian optimization, cross-validation, and executive reporting.
  • Excluded: Real-time deployment APIs, production monitoring infrastructure, deep neural network paths, adversarial robustness analysis, and other financial domains beyond credit risk.
Datasets

Interactive dataset command center

The project uses two public credit datasets with sharply different size, structure, and imbalance characteristics so the framework can be tested under both industrial-scale and compact benchmark conditions.

Dataset 1

American Express Default Prediction Dataset

This dataset includes approximately 458,913 customers and 190 anonymized features. It is an industrial-level credit risk dataset with time-based statement records, broad feature categories, high missingness, and strong computational demands.

  • Contains delinquency, spending, payment, balance, and risk indicators across repeated statement dates.
  • Requires aggregation to customer level using mean, standard deviation, minimum, maximum, and recent value logic.
  • Contains substantial missing values and category-specific data behavior that require careful treatment.
  • Feature correlation is strong across several variable groups, so collinearity control matters.

Dataset 2

GiveMeSomeCredit Dataset

This dataset contains 150,000 borrower records and 11 financial and demographic features, with a default rate of 6.7 percent. It acts as a compact benchmark dataset for explanation clarity, baseline comparison, and controlled imbalance evaluation.

  • Compact structure makes it useful for straightforward interpretation and benchmark modeling.
  • Includes strong imbalance and missing values in fields such as income and dependents.
  • Supports easier side-by-side evaluation of model quality and explanation behavior.
  • Useful for testing threshold optimization, calibration, and fairness checks in a smaller feature space.

Methodology

End-to-end execution architecture

The methodology combines structured preprocessing, class imbalance remediation, baseline benchmarking, XGBoost optimization, SHAP explainability, fairness awareness, and reproducible engineering practices.

Data Preparation

Missing value treatment

Missingness is handled through a tailored approach based on data behavior. Random missingness is addressed with multivariate imputation, while conditional missingness is handled with predictor-driven imputation and sensitivity checks.

  • MICE-based logic where appropriate.
  • Conditional treatment for feature-linked gaps.
  • Stability checks across imputation assumptions.
Feature Layer

Engineering and aggregation

Feature engineering goes beyond raw fields by building debt capacity indicators, utilization measures, delinquency severity signals, temporal trends, and aggregated borrower-level summaries.

  • AMEX time-series aggregation to customer vectors.
  • Derived ratios and interaction signals.
  • Leakage-aware encoding and standardization.
Outlier Control

Robust anomaly handling

Extreme values are capped and scaled to reduce training distortion, while implausible multivariate combinations are reviewed through anomaly detection logic.

  • IQR review and winsorization.
  • Robust scaling for continuous variables.
  • Isolation Forest for unusual combinations.
Class Imbalance

Minority default preservation

Because credit defaults are sparse, the framework combines oversampling strategies, cost-sensitive training, and threshold optimization to reduce missed defaults without losing explanation quality.

  • SMOTE and variants.
  • Cost-sensitive learning.
  • Threshold optimization and hybrid testing.
Modeling

Benchmarks and primary engine

Logistic Regression and Decision Tree provide transparent reference models, while XGBoost serves as the primary high-performance ensemble model tuned through Bayesian optimization.

  • Logistic coefficients for clarity.
  • Decision rules for simple path tracing.
  • XGBoost with 100 Bayesian search iterations.
Explainability

SHAP-first visibility layer

SHAP TreeExplainer powers global feature ranking, local case explanations, waterfall views, dependence analysis, and counterfactual insight. LIME remains optional for comparative evaluation.

  • Global importance ranking.
  • Local case-level explanation.
  • Fidelity, stability, and communicability review.
Implementation Architecture

Modular system design

  • data_loading for dataset acquisition, validation, and quality checks.
  • preprocessing for missing values, outliers, feature engineering, and imbalance handling.
  • model_training for Logistic Regression, Decision Tree, XGBoost, and Bayesian optimization.
  • explainability for SHAP, optional LIME, and counterfactual analysis.
  • evaluation for metrics, cross-validation, statistical testing, and reporting aggregation.
  • utilities for logging, result packaging, helper functions, and visualization support.
Evaluation Logic

How quality is judged

  • AUC ROC for threshold-independent ranking power.
  • Precision for false approval control.
  • Recall for loss prevention through missed-default reduction.
  • F1 Score for balance under class imbalance.
  • MCC for robust overall quality in uneven class distributions.
  • Brier Score for probability calibration and reliability.
  • Explanation fidelity, feature stability, communicability, and counterfactual validity.

Models

Comparative model stack

The modeling layer is intentionally structured so transparent benchmarks remain visible while the primary ensemble engine demonstrates measurable improvement.

Baseline 1

Logistic Regression

Used as a reference model because coefficients directly show how features affect the probability of default.

  • Coefficient-driven interpretability.
  • Natural benchmark for regulated comparison.
  • Can miss non-linear interaction effects.
Baseline 2

Decision Tree

Used for rule-based transparency because path splits can be traced step by step and presented clearly to non-technical audiences.

  • Simple rule learning.
  • Easy to visualize and explain.
  • Needs pruning to control overfitting.
Primary Engine

XGBoost

Selected as the main predictive engine because it captures complex non-linear borrower behavior with strong efficiency and strong ranking performance.

  • Regularized boosting with second-order optimization.
  • Bayesian hyperparameter tuning.
  • Best suited as the main decision engine in this project.

Metrics

Performance and explanation review

One metric alone can hide weaknesses in imbalanced lending data, so the framework uses multiple performance and explanation quality measures together.

AUC ROCRanking power

Measures how well the model ranks likely defaults above likely non-defaults without relying on one fixed decision threshold.

PrecisionApproval control

Shows how many predicted defaults are truly defaults and helps reduce unnecessary rejection decisions.

RecallLoss prevention

Shows how many actual defaults are detected, which is critical because missed defaults translate directly into loss exposure.

F1 ScoreBalanced view

Balances precision and recall into one signal, which is especially useful when the default class is much smaller.

MCCImbalance robust

Provides one of the most balanced single-number views of performance when class sizes are uneven.

Brier ScoreCalibration

Measures how reliable predicted probabilities are, which matters for pricing, limits, and risk-aware lending operations.

Risk Command Center

Operational safeguards

The project includes explicit mitigation and contingency planning so delivery remains resilient even under data, compute, explainability, or timeline pressure.

R1

Dataset accessibility or corruption

Low–Moderate
Severity

Mitigation: immediate dataset backup and integrity checksums.

Fallback: switch to alternative public credit datasets if intake fails.

R2

Inadequate compute for AMEX

Moderate
Severity

Mitigation: GPU-ready execution, data sampling optimization, and efficient preprocessing.

Fallback: CPU-only training with feature-reduced AMEX subset.

R3

XGBoost convergence failure

Low–Moderate
Severity

Mitigation: proper initialization, learning-rate scheduling, and run monitoring.

Fallback: use stable default hyperparameters as the starting configuration.

R4

Optimization inefficiency

Moderate
Severity

Mitigation: domain-informed search ranges with coarse refinement.

Fallback: expanded random search on narrowed parameter space.

R5

SHAP memory overflow

Moderate
Severity

Mitigation: TreeExplainer plus strategic subsampling for large-scale runs.

Fallback: compute SHAP on a representative sample of 10,000 rows.

R6

Imbalance mitigation underperformance

Low
Severity

Mitigation: multiple hybrid imbalance strategies and stratified validation.

Fallback: threshold optimization with conservative default capture priority.

R7

Reproducibility failure

Low
Severity

Mitigation: fixed seeds, version control, exact dependency capture, and containerization.

Fallback: define output tolerance ranges and document environment differences.

R8

Schedule disruption

Minimal
Severity

Mitigation: built-in buffer time and versioned work tracking.

Fallback: partial delivery package with clearly documented status and next steps.

R9

Feature engineering inefficiency

Low–Moderate
Severity

Mitigation: ablation studies and validation checkpoints for each major feature block.

Fallback: use raw aggregated AMEX features without expanded engineering layers.
